## Решение (Основное): Централизованная аналитика

**Описание компонентов:**
- **Агент на ферме** — локальный сервис, собирает данные с камер и сенсоров, публикует их в Kafka. Обеспечивает базовую буферизацию данных на случай потери связи.
- **Центральный сервер** — сервер в ЦОД компании, подписан на Kafka, выполняет аналитику событий, формирует тревоги и сохраняет агрегированные данные. Видео с камер не передаётся в полном объёме — только события и тревоги, чтобы снизить нагрузку на канал. Хранение данных ведётся в TimescaleDB/ClickHouse. API предоставляет данные для мобильного/веб-приложений.

#### Контекстная диаграмма
```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml

Person(operator, "Оператор фермы")
Person(engineer, "Инженеры")
Person(mobileApp, "Мобильное приложение")

System_Boundary(system, "Система мониторинга животных") {
  System(feeders, "Кормушки и поилки")
  System(cameras, "Видеокамеры")
  System(sensors, "Датчики среды")
  System(kafka, "Kafka", "Шина данных компании")
  System(tsdb, "TimescaleDB", "Хранилище временных рядов")

  System(edgeAgent, "Агент на ферме")
  System(centralServer, "Центральный сервер")
}

Rel(edgeAgent, kafka, "Публикация данных с камер и сенсоров")
Rel(kafka, centralServer, "Подписка на данные")
Rel(centralServer, tsdb, "Хранение аналитики")
Rel(centralServer, mobileApp, "REST API для операторов")
Rel(mobileApp, centralServer, "Получение оповещений")
@enduml
```

## Альтернативное решение: Локальная аналитика

**Описание компонентов:**
- **Агент с локальной аналитикой** — расширенный агент, выполняющий локальную видеоаналитику с использованием AI (YOLO, TensorRT). Публикует итоговые тревоги и агрегированные метрики в Kafka.
- **Центральный сервер (агрегация)** — облегчённый сервер, агрегирующий события и предоставляющий API доступа.

#### Контекстная диаграмма
```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml

Person(operator, "Оператор фермы")
Person(engineer, "Инженеры")
Person(mobileApp, "Мобильное приложение")

System_Boundary(system, "Система мониторинга животных") {
  System(feeders, "Кормушки и поилки")
  System(cameras, "Видеокамеры")
  System(sensors, "Датчики среды")
  System(kafka, "Kafka", "Шина данных компании")
  System(tsdb, "TimescaleDB", "Хранилище временных рядов")

  System(edgeAgent, "Агент с локальной аналитикой")
  System(centralServer, "Центральный сервер (агрегация)")
}

Rel(edgeAgent, kafka, "Публикация тревог и метрик")
Rel(kafka, centralServer, "Получение агрегированных событий")
Rel(centralServer, tsdb, "Хранение метрик")
Rel(centralServer, mobileApp, "REST API")
Rel(mobileApp, centralServer, "Просмотр тревог")
@enduml
```

### Недостатки, ограничения, риски

**Основное решение:**

- Зависимость от качества связи и Kafka;
- Плюс — переиспользуем корпоративную инфраструктуру;

**Альтернативное решение:**

- Увеличенная стоимость edge;
- Плюс — автономность ферм без интернет;
- Все данные всё равно синхронизируются с Kafka для единой аналитики компании.

---

## Коррекция схем взаимодействий
- Оператор фермы напрямую не взаимодействует с IT-агентами.
- Все взаимодействия операторов идут через централизованный API (мобильное приложение).
- Локальные оповещения считаются физическими средствами (сирены, световые панели) и не моделируются в C4.
