# ADR: Выбор архитектуры MVP — Локальная аналитика

### Название задачи:
Анализ и обоснование выбора локальной аналитики для MVP мониторинга животных

### Автор:
Шаханов Михаил

### Дата:
17 июля 2025

---

### Функциональные и нефункциональные требования
Все требования перечислены в Task1. Ключевой фокус — устойчивость к отсутствию интернета, минимальная латентность тревог, автономность ферм.

---

### Решение
Выбран вариант с локальной видеоаналитикой на edge-агентах с передачей агрегированных метрик и тревог через Kafka. Централизованный сервер агрегирует данные и предоставляет API для внешних систем. Данные хранятся в TimescaleDB и ClickHouse.

```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml
Person(operator, "Оператор фермы")
System(system, "Система мониторинга животных") {
  System(edgeAgent, "Edge Agent", "Локальная видеоаналитика, MQTT")
  System_Ext(kafka, "Kafka", "Брокер данных компании")
  System_Ext(timescale, "TimescaleDB", "Хранилище временных рядов")
  System(centralServer, "Центральный сервер", "Агрегация тревог, API")
}
Rel(edgeAgent, operator, "Получает оповещения")
Rel(edgeAgent, kafka, "Отправляет тревоги и метрики")
Rel(kafka, centralServer, "Передача данных")
Rel(centralServer, timescale, "Сохраняет данные")
@enduml
```

---

### Преимущества
- Минимальная задержка тревог, автономность на фермах;
- Устойчивость при плохом интернете;
- Быстрая реакция AI локально;
- Возможность оффлайн работы.

### Недостатки
- Более дорогие edge-агенты с AI;
- Сложность поддержки в полях;
- Требуется больше компетенций локального персонала;
- Выше затраты на инфраструктуру.

---

### Вывод
Локальная аналитика рекомендована для ферм с проблемной связью и высокой ценностью быстрого реагирования. Для MVP проект предлагается в качестве параллельной пилотной ветки с ограниченным масштабом.
