# ADR: Выбор архитектуры MVP — Централизованная аналитика

### Название задачи:
Анализ и обоснование выбора централизованной архитектуры для MVP мониторинга животных

### Автор:
Шаханов Михаил

### Дата:
17 июля 2025

---

### Функциональные и нефункциональные требования
Все требования перечислены в Task1. Ключевой фокус — скорость вывода MVP, минимальные инвестиции на старте, использование существующей корпоративной инфраструктуры.

---

### Решение
Используется централизованная архитектура с публикацией данных в Kafka, обработкой событий на Центральном сервере и хранением данных в TimescaleDB и ClickHouse. Централизованный сервис выполняет дополнительную видеоаналитику, формирует тревоги и предоставляет API для внешних систем.

```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml
Person(operator, "Оператор фермы")
System(system, "Система мониторинга животных") {
  System_Ext(kafka, "Kafka", "Брокер данных компании")
  System_Ext(timescale, "TimescaleDB", "Хранилище временных рядов")
  System(centralServer, "Центральный сервер", "События, аналитика, API")
}
Rel(centralServer, operator, "Получает оповещения")
Rel(centralServer, kafka, "Получает данные с агентов")
Rel(centralServer, timescale, "Сохраняет данные")
@enduml
```

---

### Преимущества
- Быстрое внедрение MVP, минимум нового кода;
- Простая поддержка централизованной аналитики;
- Полная интеграция с Kafka и BI компании;
- Упрощённое масштабирование на другие фермы.

### Недостатки
- Зависимость от качества интернета;
- Возможны задержки в тревогах;
- Требуется стабильный Kafka кластер.

---

### Вывод
Централизованная архитектура рекомендована для MVP как оптимальный компромисс между стоимостью, скоростью внедрения и интеграцией с корпоративной платформой.
